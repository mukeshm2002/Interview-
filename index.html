1. Java (Core, OOPs, Collections & DSA)

அடிப்படை ஜாவா (Core Java)

An array is a **fixed-size** data structure used to store multiple values of the same data type. I understand its concept (single/multi-dimensional), but I **did not use raw arrays** in my projects.

Instead, I used **ArrayList** and **LinkedList**. The reason is that arrays require you to specify a fixed size, which is inefficient when dealing with dynamic data like user lists or orders. Collections like ArrayList offer **flexible, dynamic resizing**, which is much cleaner and requires less boilerplate code to manage varying numbers of elements.

Class: A template or blueprint that defines properties (attributes) and behaviors (methods).

Object: An instance of a class, representing a specific entity with the properties and behaviors defined by the class.

Example:

Imagine a class called "Dog". This class defines the general characteristics of all dogs, such as breed, age, and bark(). An object would be a specific dog, like "Buddy", with its own breed, age, and the ability to bark.

  • public: Accessible from any class.
  • private: Accessible only within the same class.
  • protected: Accessible within the same package and subclasses.
  • default (package-private): Accessible within the same package.

String: **Immutable** (cannot be changed after creation).

StringBuilder: **Mutable** and is generally **faster** than StringBuffer (Non-synchronized).

StringBuffer: **Mutable** and is **Thread-Safe** (synchronized).

String objects cannot be changed after creation. This is critical for **Security** (used heavily in class loading), **Thread Safety** (multiple threads can share String objects without synchronization issues), and **Efficiency** (it allows for String Pool optimization).

final: A keyword used to make a variable, method, or class **unchangeable**.

finally: A block that **always executes** after a try-catch block, regardless of whether an exception occurred.

finalize: A method called by the **garbage collector** before an object is reclaimed (often discouraged due to unpredictable timing).

OOPs கான்செப்ட்ஸ் (Object-Oriented Programming)

Encapsulation: Wrapping data and methods that operate on the data into a single unit (a class), and restricting access to some of the object's components.
EX: A "capsule" medicine, where the medicine is enclosed in a shell, making it safe and accessible only when needed.

Abstraction: Hiding complex implementation details and showing only the essential information.
EX: A "smartwatch" displaying time and heart rate, without showing the internal mechanisms.

Inheritance: A class inheriting properties and methods from another class.
EX: Creating "Apple" and "Orange" classes from a "Fruits" class, where both inherit common fruit properties.

Polymorphism: An action working in different ways.
EX: A "cooking" action that can "stir" for vegetables and "fry" for meat.

Abstract Class:

  • Can have abstract and non-abstract methods.
  • Can have instance variables.
  • Classes **extend** an abstract class.
  • Provides partial implementation.

Interface:

  • All methods were abstract (before Java 8).
  • Cannot have instance variables (only constants).
  • Classes **implement** an interface.
  • Provides a contract for what a class must do.

Method Overloading: Defining multiple methods with the same name but different parameters in the same class. Occurs at **compile time** (Static Polymorphism).

Method Overriding: Subclass redefines the parent class method with the same signature. Occurs at **runtime** (Dynamic Polymorphism).

கலெக்‌ஷன்ஸ் & ஸ்ட்ரீம் API (Collections & Stream API)

I utilized the following Java Collections to efficiently store and manipulate data across my projects:

  • ArrayList: Used to handle lists of entities retrieved from the database (e.g., fetching a list of available bus routes).
  • LinkedList: Used for scenarios requiring frequent additions or removals, such as managing a sequence of historical patient visits.
  • HashMap: Used for storing data as Key-Value pairs to enable quick lookups based on unique IDs (e.g., mapping Employee ID to the Employee object).
  • HashSet: Used to maintain a collection of unique items, such as storing a list of distinct booked seat numbers to prevent duplicate bookings.

It is a **unified architecture** that simplifies storing, manipulating, and retrieving a group of objects (data).

  • The three core interfaces are List (for ordered, index-based access), Set (for unique elements/no duplicates), and Map (for fast Key-Value storage).
  • In my projects, I rely on ArrayList to handle entity lists and HashMap for quick lookups based on unique IDs.

Both collections store lists, but they differ in how they work internally:

  • ArrayList: Uses a dynamic array. Accessing elements by index (Search) is **fast ($\text{O}(1)$)**. Insertion/Deletion from the middle is **slow ($\text{O}(n)$)**.
  • LinkedList: Uses a doubly-linked list. Insertion/Deletion is **fast ($\text{O}(1)$)**. Accessing an element by index (Search) is **slow ($\text{O}(n)$)**.

The Stream API was introduced in **Java 8** to process collections in a declarative, functional style.

It allows operations like **filtering**, **mapping**, and **reducing** data in a chain (pipelining) without modifying the source collection.

Functional Interface: An interface with only **one abstract method** (e.g., `Runnable` or `Comparator`).

Lambda Expression: Provides a **concise way** to implement the single abstract method of a functional interface, making code shorter and more readable.

Default Method: Allows adding new methods to interfaces without breaking existing implementations. Can be **overridden** by the implementing class.

Static Method: A utility method tied directly to the interface, which **cannot be overridden** and must be called using the interface name.

விதிவிலக்கு கையாளுதல் (Exception Handling)

Exception handling is a mechanism to handle runtime errors in a program.

In my Bus Reservation System, I used **custom exceptions** to handle specific errors.

  • NoRouteFoundException: When a route is not found.
  • BusNotFoundException: When no buses are available on a specific date.
  • InsufficientSeatException: When there are not enough seats available.
  • SeatAlreadyBookedException: When a seat is already booked.

Checked Exceptions: Checked at compile time (e.g., IOException, SQLException). Must be explicitly handled using a try-catch block or declared using throws.

Unchecked Exceptions (Runtime): Occur at runtime (e.g., NullPointerException, ArrayIndexOutOfBoundsException). Handling is optional, as they often indicate programming errors.

throw: Used to explicitly **throw an exception** within a method.

throws: Used in a method signature to **declare** which exceptions a method might throw.

DSA (டேட்டா ஸ்ட்ரக்சர்ஸ் & அல்காரிதம்ஸ்)

These are graph traversal algorithms:

BFS (Breadth-First Search): Explores all neighbors at the present level before moving to the next level (level by level, uses a **Queue**).

DFS (Depth-First Search): Explores as far as possible along each branch before backtracking (branch by branch, often uses a **Stack** or recursion).

Binary Search Tree (BST): A binary tree data structure where the value of every node in the left subtree is less than the value of the root node, and the value of every node in the right subtree is greater. This allows for fast searching and sorting.

Stack: A linear data structure that follows the **LIFO** (Last In, First Out) principle.

Queue: A linear data structure that follows the **FIFO** (First In, First Out) principle.

DSA is about selecting the most efficient way to organize and process data to optimize time and memory.

I applied DSA concepts by choosing **HashMap** for fast O($$1$$) lookups of entities (e.g., fetching a patient record by ID) and **ArrayList** for ordered, indexed lists, prioritizing efficient access based on the project's specific needs.

2. Spring & Spring Boot (Backend)

அடிப்படை ஸ்பிரிங் பூட் (Core Spring Boot)

DI is a design pattern used by Spring where the **creation and management of object dependencies** are handled by the Spring IoC (Inversion of Control) container, not the object itself. This promotes **loose coupling** and makes Unit Testing with Mockito much easier.

PROJECT USE: I use @Autowired or constructor injection to get an instance of my Service layer inside my Controller.

Spring Boot is highly modular. The core modules I utilized in my projects include:

  • **Spring Web:** For building RESTful APIs (Controllers).
  • **Spring Data JPA:** For database interaction and ORM.
  • **Spring Security:** For authentication and authorization.

I connected Spring to the database (**MySQL** in most cases) using **Spring Data JPA**. I configured the database connection details (URL, username, password) in the `application.properties` file.

Spring Data JPA simplifies database access by providing a high-level abstraction. It allows me to define repositories and use simple method names (like `findByUsername()`) instead of writing repetitive, boilerplate **CRUD** (Create, Read, Update, Delete) SQL queries.

Apache Tomcat is the **default embedded web server and servlet container** used by Spring Boot. It allows the Spring application to run as a standalone Java application without needing external server installation.

ஸ்பிரிங் MVC (Spring MVC)

The MVC (Model-View-Controller) structure separates concerns:

  • **Model:** Holds the application data (e.g., the `Employee` object).
  • **View:** The presentation layer (e.g., the HTML page built with Thymeleaf).
  • **Controller:** Handles incoming requests, processes input, interacts with the Service layer, and chooses the appropriate View.

The **Dispatcher Servlet** acts as the Front Controller in Spring MVC. It is the first component to receive every incoming web request, and its job is to delegate that request to the appropriate Controller method based on the URL mapping.

The View layer is responsible for presenting data to the user. In my projects, I used **Thymeleaf** as the View layer. The Controller passes the Model data to the View, which then renders the final HTML page the user sees.

Thymeleaf is a popular **server-side template engine** used to create dynamic HTML pages in Spring MVC. It allows me to inject dynamic data from the Java backend (Model) directly into the HTML structure.

REST API & டெஸ்டிங் (REST API & Testing)

RESTful APIs are web services built using the **HTTP protocol** rules. They are stateless and use standard HTTP methods to perform CRUD operations on resources:

  • **POST** to Create (e.g., create a new patient).
  • **GET** to Read/Retrieve (e.g., get all employees).
  • **PUT/PATCH** to Update.
  • **DELETE** to Remove.

@Controller: Used for traditional Spring MVC applications that typically return a view name (like a Thymeleaf template).

@RestController: A convenience annotation that combines @Controller and @ResponseBody. It is used exclusively for building **RESTful Web Services** and returns data directly (usually JSON/XML).

Mockito is a popular **mocking framework** used for writing effective **unit tests**. I used Mockito to create mock objects of dependencies (like the Repository layer) so I could test the logic of my Service layer in isolation without actually hitting the database.

பாதுகாப்பு & வெளியீடு (Security & Deployment)

Spring Security is a framework that provides powerful **Authentication** (verifying who you are) and **Authorization** (verifying what you can do). I used it to secure API endpoints, ensuring only logged-in users could access specific resources in the EMS or Patient Management systems.

Render is a PaaS (Platform as a Service) solution I used for **building and deploying** my full-stack applications. It simplifies the deployment process by automatically building the Spring Boot application from my Git repository and managing the server environment.

3. Frontend (HTML, CSS, JavaScript)

HTML & CSS Core

Semantic elements are HTML tags that clearly describe their meaning to both the browser and the developer. Examples I use are **`<header>`**, **`<footer>`**, **`<nav>`**, and **`<section>`**. They improve accessibility and SEO.

It defines how an element's total space is calculated, consisting of four parts, from inside out:

  1. **Content:** The actual element size (width/height).
  2. **Padding:** Space between the Content and the Border.
  3. **Border:** The line surrounding the element and its padding.
  4. **Margin:** Clear space outside the Border, separating the element from others.

I use box-sizing: border-box; to make width/height include padding and border, simplifying layout calculations.

CSS properties are the styling rules applied to HTML elements. Key properties I use are **`color`**, **`margin`** and **`padding`**, and especially **`display`** (using `flex` or `grid`) for responsive layout management.

Flexbox: A **1-dimensional** layout system (rows OR columns). Great for aligning items inside a component, like navigation links or form elements.

CSS Grid: A **2-dimensional** layout system (rows AND columns simultaneously). Used for creating the overall page layout structure (e.g., header, sidebar, main content).

Media Queries are a CSS feature that allows applying styles only when a certain condition is met, most commonly the device's screen size. This is essential for **Responsive Design**.

PROJECT USE: I used them to hide or rearrange the sidebar navigation when the screen width drops below a tablet size (@media (max-width: 768px)), providing a mobile-friendly experience.

JavaScript & DOM Manipulation

The **DOM (Document Object Model)** is the tree-like structure created by the browser to represent the HTML document. It's the programming interface that allows JavaScript to interact with, modify, and update the content and structure of the HTML page dynamically.

These are the two ways events propagate through the DOM tree:

  • **Capturing (Top-Down):** The event starts at the window and travels down to the target element.
  • **Bubbling (Bottom-Up):** The event starts at the target element and travels up to the window. **Bubbling is the default** behavior in most browsers.

**Event Delegation** uses the **Bubbling** phase by placing one listener on a parent element to manage events for all its children, which is great for performance.

A closure is the ability of an **inner function** to **access the variables of its outer function** even after the outer function has finished executing.

It's used to create **private variables** and maintain state. In my projects, I use it when generating dynamic form elements to ensure each element retains its correct, private ID for handling data submission.

  • **var:** **Function-scoped**. Can be redeclared and updated. **Hoisted**. Avoided in modern JS.
  • **let:** **Block-scoped** (inside {}). Cannot be redeclared but can be updated.
  • **const:** **Block-scoped**. Cannot be redeclared **or** updated (must be initialized). Used for values that should not change.

I prioritize const by default, then let only if a variable needs reassignment, and I avoid var entirely for clean code.

Asynchronous Operations (The Core of Frontend)

**AJAX (Asynchronous JavaScript and XML)** is the technique of making requests to a server **without reloading the page**. Fetch is the modern standard for this.

PROJECT USE: I use fetch() to call my Spring Boot REST APIs (e.g., fetch('/api/routes')). It returns a **Promise**, which I handle using **.then()** to process the JSON data and update the DOM.

A Promise is an object representing the **eventual completion (or failure) of an asynchronous operation**. It acts as a placeholder for a value that will become available in the future.

It has three states: **Pending** (initial), **Fulfilled** (successful completion), or **Rejected** (failed). It helps manage asynchronous flow cleanly, avoiding **Callback Hell**.

Both are Web Storage APIs for storing key-value pairs locally in the browser.

  • **localStorage:** Data **persists** even after the browser is closed. PROJECT USE: Storing user preferences or theme settings.
  • **sessionStorage:** Data is cleared when the **browser tab/window is closed**. PROJECT USE: Storing temporary data like a multi-step form's current progress or a logged-in user's temporary state.
