Class & Object :

  • In Java, a Class is like a blueprint or template that defines how an object should be ‚Äî it contains variables and methods.
  • An Object is an actual instance of a class ‚Äî it‚Äôs created from that blueprint and has real values.

OOPS 4 Pillars :

  • Encapsulation : Wrapping (or combining) data (variables) and methods (functions) that operate on that data into a single unit (class).
  • Inheritance : A class (child) can inherit the properties and methods of another class (parent).
  • Polymorphism : The same method can behave differently based on the object calling it. Types: Compile-time (Method Overloading), Runtime (Method Overriding)
  • Abstraction : Showing only the important details and hiding the implementation.

Method Overloading and Method Overriding :

  • Method Overloading happens within the same class, when multiple methods have the same name but different parameters (different type, number, or order). It‚Äôs a compile-time polymorphism.
  • Method Overriding happens in different classes ‚Äî a child class provides a new version of a method that already exists in its parent class. It‚Äôs a runtime polymorphism.

Abstract class vs Interface :

An abstract class is used when we want to share common behavior among subclasses, and it can have both abstract and normal methods.
An interface defines a contract that classes must follow ‚Äî it supports multiple inheritance and achieves full abstraction.
In my Bus Booking Project, I used interfaces like BookingService and PaymentService to define structure, and abstract classes like User for shared properties between Admin and Customer.

Throw and Throws :

  • The throw keyword is used to manually throw an exception inside a method or block of code. It‚Äôs followed by an object of an exception class (like new Exception()).
  • The throws keyword is used in a method declaration to tell the compiler that this method might throw an exception, and it should be handled by the calling method.

Design Pattern :

Design Patterns are proven solutions to common software design problems. They help make the code more reusable, maintainable, and scalable.

There are three main types of design patterns:

Spring Boot :

Spring Boot is a framework built on top of the Spring Framework. Its main goal is to make it easier and faster to start and develop Spring applications.
It removes the need for lots of manual configuration, so developers can focus on writing business logic.

The main idea of Spring Boot is ‚ÄúConvention over Configuration‚Äù, meaning it uses smart defaults so we don‚Äôt have to set up everything ourselves.
For example, if we add spring-boot-starter-web, it automatically sets up everything needed for a web app ‚Äî like the Dispatcher Servlet and other MVC parts.
Another big advantage is embedded servers like Tomcat or Jetty. They let us run the app directly as a single JAR file, without installing a separate server.

Spring MVC :

Spring MVC is a main part of building web applications in Spring. MVC stands for Model-View-Controller, and it helps to separate the logic, design, and user input of an application.
When a user sends a request, the Controller receives it. The Controller then calls the Model, which handles the business logic and data. After processing, the Model sends the data back to the Controller. Finally, the Controller passes that data to the View (like a Thymeleaf page or JSON response) to show it to the user.
This structure keeps the code clean, easy to test, and simple to maintain.
For example, in my Employee Management System, the Controllers handled API requests, the Services managed the business logic, and the Repositories connected with the database.

View Layer :

In Spring MVC architecture, the View layer is the part that shows data to the user ‚Äî it‚Äôs what users actually see and interact with, like employee details, bus routes, or appointment timings.
The View layer can be built using different technologies. In my projects, such as the Employee Management System, I used Thymeleaf for the View layer. Thymeleaf is a server-side template engine that helps us add dynamic data into HTML pages.
After the Controller gets data from the Model, it chooses which Thymeleaf template to use and passes the data to it. Then Thymeleaf processes that data and creates the final HTML page, which is sent to the user‚Äôs browser. This setup helps to keep the design (view) separate from business logic, while still allowing dynamic web pages.

Rest API :

In Spring Boot, a RESTful API is used to build communication between the client and server using HTTP methods. REST stands for Representational State Transfer, and it follows a simple rule ‚Äî use standard HTTP methods for operations. For example:
  • GET ‚Üí to fetch data
  • POST ‚Üí to add new data
  • PUT ‚Üí to update existing data
  • DELETE ‚Üí to remove data
In Spring Boot, we create REST APIs using the @RestController annotation. Inside that, we map URLs to methods using annotations like @GetMapping, @PostMapping, @PutMapping, and @DeleteMapping. The methods usually return JSON data, which the client (like React, Angular, or mobile app) can easily use. For example, I created REST APIs in my Employee Management System, where the controller handled employee CRUD operations ‚Äî like adding, viewing, updating, and deleting employee details. This makes the application lightweight, scalable, and easy to integrate with any frontend or external system.

Database Connectivity :

Connecting Spring Boot to a database is very easy with Spring Data JPA.
First, we add the spring-boot-starter-data-jpa dependency. It includes everything we need, including Hibernate, which helps map Java classes to database tables.

Next, we set up the database connection in the application.properties file. We just add the database URL, username, and password, and Spring Boot automatically creates the connection for us.
Then, we create our data models as JPA entities. For example, an Employee class marked with @Entity will be linked to an employee table in the database.

Finally, we make repository interfaces by extending JpaRepository. This gives us all the basic CRUD operations ‚Äî like save, find, update, and delete ‚Äî without writing extra code.
This makes database handling simple and fast, so we can focus more on our business logic.

Spring Security :

Spring Security is a powerful framework that provides authentication and authorization for Spring applications.
It helps protect applications by ensuring that only the right users can access specific parts of the system.
Authentication means verifying who the user is ‚Äî usually through a username and password.
Authorization means checking what the user is allowed to do ‚Äî like accessing certain APIs or pages.
In Spring Boot, adding spring-boot-starter-security automatically sets up basic security. It secures all endpoints and provides a default login page.
You can customize it by defining user roles, using a database for user details, or securing APIs with JWT (JSON Web Token) for stateless authentication.
Overall, Spring Security makes it simple to build secure, role-based, and token-based authentication systems in Spring Boot applications.

Spring Thymeleaf :

Spring Thymeleaf is a server-side template engine used in Spring Boot applications to create dynamic web pages.
It allows you to combine HTML with Spring data ‚Äî meaning you can easily show data from your backend directly in your web pages.
The Controller sends data (like employee details or user info) to the Thymeleaf template.
In the template, we use simple expressions like ${employee.name} to display that data inside the HTML.
When the page is rendered, Thymeleaf replaces those placeholders with real data and sends the final HTML to the browser.
Thymeleaf works well with Spring MVC and supports form handling, loops, conditions, and internationalization.
It‚Äôs easy to read, looks like normal HTML, and helps keep your frontend and backend connected smoothly while building dynamic web apps.

Test Cases Integration :

In my project, I used Mockito for writing unit test cases in Spring Boot. Mockito helps me test the logic of a class without using real dependencies like a database or external API.
For example, if I‚Äôm testing a Service class, I can create a mock repository using the @Mock annotation, and inject it into my service using @InjectMocks. Then, I can define how the mock should behave ‚Äî like returning some fake data when a method is called.
This way, I can focus only on testing my business logic, not the real database. It makes testing faster, easier, and more reliable.
I usually write these test cases using JUnit along with Mockito, to check if my methods return the correct results or handle exceptions properly.

DOM manipulation :

Event loop in JavaScript :

Git & Github :

Git is a distributed version control system that lets developers track changes in their code and collaborate on projects. GitHub is a web-based platform that provides hosting for Git repositories, along with features for project management, collaboration, and code review.
  • git init: This command creates a new Git repository in the current folder.
  • git clone: This command creates a copy of a remote Git repository on your local machine.
  • git status: This command shows the current state of your repository, including which files have been modified or staged.
  • git add: This command adds files to the staging area, preparing them for the next commit.
  • git commit: This command creates a snapshot of your project's changes, saving them in the repository's history.
  • git push: This command sends your committed changes from your local repository to a remote repository.
  • git pull: This command fetches and merges changes from a remote repository to your local one.
  • git branch: This command lets you manage branches in your repository, such as creating, listing, or deleting them.
  • git checkout: This command lets you switch between branches or restore files from a specific commit.
  • git merge: This command combines the changes from one branch into another.
