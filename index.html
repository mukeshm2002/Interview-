My Interview Prep Guide

1. Java (Core & OOPs)

**Key Points:** Arrays store **multiple values of the same data type**. Elements are accessed using an **Index**, which starts at **0**.

**My Project Link:** *I mostly used **Lists** in my projects for dynamic sizing, but the core concept of an array is essential for understanding how to manage fixed-size collections efficiently.*

**Overloading:** **Same** method name, **Different** arguments (in the same class). **Overriding:** **Same** method name and arguments, in the **Child Class** (to provide specific logic).

**My Project Link:** *I used **Overriding** heavily in my Service layer where a child service class might implement a specific behavior defined by a parent interface or class.*

**Key Points:** It handles **runtime errors** without crashing the application. I used **`try-catch`** blocks to manage errors like **`ResourceNotFoundException`** (e.g., in EMS if an employee ID is invalid) and database connection issues.

**My Project Link:** *I used a global **`@ControllerAdvice`** in the Spring Boot projects to handle all exceptions uniformly and return meaningful error messages (JSON) to the frontend.*

**E.I.P.A.:** **Encapsulation** (Hiding data using private fields), **Inheritance** (Code reuse), **Polymorphism** (One name, many forms), **Abstraction** (Showing only essential info using interfaces/abstract classes).

**My Project Link:** *I applied **Encapsulation** in all my entities (Employee, Patient, Bus) and used **Abstraction** by defining Service layer interfaces before writing the implementation.*

**Stream API:** Used for **efficiently processing** collections (like filtering or mapping data). **Lambda:** A **short block of code** used to implement Functional Interfaces, making Stream API concise.

**My Project Link:** *In the **Bus Reservation System**, I used the Stream API to **filter** the list of available buses based on the user's search criteria (route and date) before displaying them.*

2. Spring & Spring Boot

**Key Modules:** **Spring Web** (for building REST APIs), **Spring Data JPA** (for DB connectivity/CRUD), and **Spring Security** (for user roles/login).

**My Project Link:** *In the **EMS and Patient Systems**, I primarily used **Spring Web** to handle API requests and **Spring Data JPA** to manage all data (employees, appointments, medicine).*

**What:** A set of rules for building web services based on the **HTTP protocol**. It uses standard HTTP methods (**GET, POST, PUT, DELETE**).

**My Project Link:** *My **EMS** and **Patient Management** systems are fully built on REST APIs. For example, to get a list of all employees, I use a **GET** request to the `/api/employees` endpoint.*

**What:** It simplifies database access by eliminating boilerplate code. We only define **Repository Interfaces**, and JPA automatically handles **CRUD** operations.

**My Project Link:** *I created **`EmployeeRepository`** and **`PatientRepository`** interfaces, which allowed me to save, find, and delete records without writing any SQL or complex JDBC code.*

**Purpose:** Provides **Authentication** (verifying identity/login) and **Authorization** (controlling access to resources based on roles).

**My Project Link:** *Although my full **Role-Based Access** (Admin/Passenger) for the Bus system is a work in progress, I understand Spring Security is necessary to protect the Admin endpoints (like adding a bus/fixing the price).*

**Key Points:** It acts as the **Front Controller** in Spring MVC. It receives **ALL** incoming requests and dispatches them to the correct Controller method for processing.

**My Project Link:** *This is the entry point for all API calls in my projects, responsible for making sure a request (e.g., to create a new employee) reaches the right method.*

3. Frontend (HTML/CSS/JS)

**What:** HTML tags that **clearly describe their meaning** (e.g., **`header`**, **`footer`**, **`nav`**, **`article`**). **Benefit:** Improves **SEO** and **Accessibility**.

**My Project Link:** *I used semantic tags like **`nav`** for the main menu and **`section`** to clearly define the different content areas in my EMS list view page.*

**What:** **Document Object Model**. It is a **Tree-like representation** of an HTML document in the browser's memory. **Role:** JavaScript uses the DOM to **access** and **manipulate** the content.

**My Project Link:** *When a user clicks 'Edit' on an employee row, JavaScript uses the DOM to find the specific employee data and display it in the edit form.*

**Key Properties:** **`display`** (especially **`flex`** or **`grid`** for layout), **`margin`**, **`padding`**, **`color`**, and **`font-size`**.

**My Project Link:** *I primarily use the Bootstrap framework, which relies on **Flexbox** (via utility classes) and various spacing properties to ensure the forms and lists look good on mobile.*
